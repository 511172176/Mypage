3）若錄取，學習時間如何安排？ #

  • 平日：每日 2–3 小時（1h 基礎&文件閱讀、1h 實作、0.5–1h 紀錄/反思）。
  • 週末：各 4 小時專題衝刺（功能切片→上線→回顧）。
  • 節奏：以「每週可 demo」為最小單位，持續整合（CI）與最小可行產品（MVP）。
  • 回饋：每週撰寫學習週誌與待辦分解，確保可追蹤與可交付。

4）技術日新月異，如何選擇投入領域？ #

  1. 以使用者價值為準：是否能替特定族群「省時、省錢、降風險」。
  2. 以可遷移能力為本：資料結構、網路、系統與測試等通用底層。
  3. 以小步快跑驗證：4 週內做出 MVP 上線收回饋，若無 traction 就轉向。
  4. 以安全與合規思維：尤其區塊鏈/AI 產品，預先納入風險控管。

5）一次負面情緒經驗與處理 #

在整合 ZK 驗證與合約時，曾因驗證失敗（AA23 reverted）反覆卡關，壓力與挫折感明顯。我的處理：
① 把問題拆成最小可重現情境（簡化合約與 proof）；② 檢查工具鏈版本、撰寫單元測試；③ 向社群與同儕請益，記錄排錯日誌；④ 安排運動與睡眠，避免情緒淹沒理性。最終定位錯誤來源後，建立 Debug Checklist，後續同類問題可快速排除。

6）本頁開發的技術心得 #

  • 純 HTML + CSS + 少量 JS 完成單檔網站，方便部署到 GitHub Pages。
  • 使用 scroll-margin-top + sticky header 改善錨點體驗。
  • 以 color-mix() 與 oklab 做柔和深淺主題；支援 prefers-color-scheme。
  • 版面採 12 欄格線與流體字級（clamp()），在手機也易讀。

7）如何看待 AI 對軟體工程師的影響？ #

AI 將持續拉高生產力基準：需求理解、架構設計、測試與維運仍需工程師負責。我的策略是把 AI 視為「自動化夥伴」，用於產生雛形、撰寫測試、重構與文件；同時強化對系統邏輯與風險的把關能力，避免「會跑但不可靠」的技術債。
